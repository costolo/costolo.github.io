Blocks, Procs, and Lambdas

oh my!

22 february 2015

Blocks, procs, and lambdas share many similarities but also many differences. At their core, each of the three are a coded set of instructions. To start things off, let's take a look at the syntax of the three.

blocks

[1,2,3,4].map { |x| p x*x }

Blocks are not much more than code that is passed to a method. They aren't objects, so they can't be called themselves.

(1..100) { |x| x*3 }

The above code would result in a syntax error, as there's no method that the block is being called on.

procs

proc = Proc.new { |x| p x*x} (1..10).each(&proc)

This would print out the results of running the proc on the range 1 through 10. And because procs are objects we can create instances of them for ease of use and to code more DRY-ly. Another benefit of procs is that you can pass multiple procs to a method, as with blocks you are limited to only one.

lambdas

l = lambda {|x| p x*x} (1..10).each(&l)

Lambdas and procs are more similar to each other than they are to blocks. They are actually both instances of the Proc class. One fundamental difference between the two is that lambdas will check the number of arguments passed to them. A proc with too many arguments passed will only execute the code on the first argument, and not the others. A lambda with the incorrect number of arguments will throw an argument error.

To close things off, blocks, procs and lambdas function similarly, but have nuanced differences. For a more in depth look, take a look at Adam Waxman's blog post about the differences between the three.
